## 1. Executive Summary

### What Is Juggernaut?

A **modular, plugin-based desktop application** for managing WordPress content. It syncs WordPress posts to a local SQLite database, lets you bulk-edit them offline with a powerful table/modal interface, and pushes changes back with conflict detection. A plugin system extends support for WordPress ecosystem plugins (MetaBox, SEOPress, ACF, Yoast, WooCommerce, etc.).

### Fresh Start vs Fix?

**Recommendation: Fix the existing app.** Here's why:

#### Why Not Start Fresh?

Starting fresh means rewriting **~9,500 lines** and losing 2-3 weeks to get back to the current state. The target architecture is essentially what's already partially built. The plugin type system alone (696 lines of carefully designed interfaces) would take significant effort to recreate.

The broken parts are concentrated in the **UI layer** ÔÇö specifically 3 component files and 2 config files. The fix path is clear: make UI components consume the plugin/profile system that already exists but isn't wired up to the frontend.

### Key Principle: The Core Knows Nothing About Specific WordPress Plugins

The core app handles: posts, pages, custom post types, taxonomies, basic fields (title, content, excerpt, featured image, status, slug), local SQLite caching, dirty-state tracking, push/pull sync.

**Everything else comes from plugins.** MetaBox fields, SEOPress SEO data, ACF groups, WooCommerce products ÔÇö all via the plugin system.

## 3. Plugin System

### 3.1 Plugin Interface

Already defined in `src/lib/plugins/types.ts`. Every plugin implements `JuggernautPlugin`:

```typescript
interface JuggernautPlugin {
  // Metadata
  id: string;
  name: string;
  version: string;
  manifest: PluginManifest;

  // Lifecycle
  initialize(core: CoreAPI): Promise<void>;
  activate(profile: SiteProfile, settings: Record<string, unknown>): Promise<void>;
  deactivate(): Promise<void>;

  // Data transformation (WP Ôćö Local)
  transformResourceForSync?(resource: WPResource): Promise<WPResource>;
  transformResourceForPush?(resource: LocalResource, payload: PushPayload): Promise<PushPayload>;

  // UI Extensions ÔÇö THIS IS THE KEY MISSING PIECE
  getTabs?(): TabDefinition[];
  getFieldRenderers?(): Record<string, ComponentType<FieldRendererProps>>;
  getSettingsPanel?(): ComponentType<SettingsPanelProps>;
  getFilterComponents?(): ComponentType<FilterComponentProps>[];

  // WordPress detection
  detectWordPressPlugin?(baseUrl: string, authHeader: string): Promise<boolean>;
}
```

### 3.2 What's Implemented vs What's Missing

| Capability | Interface Defined | Backend Implemented | Frontend Wired Up |
|------------|:-:|:-:|:-:|
| Plugin lifecycle (init/activate/deactivate) | Ôťů | Ôťů | N/A |
| Data transform hooks (sync/push) | Ôťů | Ôťů | N/A |
| `getTabs()` ÔÇö plugin-provided UI tabs | Ôťů | ÔŁî Not called | ÔŁî EditModal ignores it |
| `getFieldRenderers()` ÔÇö generic field rendering | Ôťů | ÔŁî Not called | ÔŁî Fields hardcoded |
| `getSettingsPanel()` | Ôťů | ÔŁî Not called | ÔŁî |
| `getFilterComponents()` | Ôťů | ÔŁî Not called | ÔŁî |
| WordPress plugin detection | Ôťů | Ôťů MetaBox only | ÔŁî No first-launch wiring |

**The type system is there. The backend hooks work. The frontend doesn't use any of the UI extension points.** This is the core gap.

### 4.3 What Needs to Happen

1. `site-config.ts` should read site targets from the active profile's `sites` array, not from a hardcoded const

3. When a profile is imported, its `required_plugins` should auto-enable matching plugins

### 7.2 Known Issue: Native Module ABI Mismatch

**Root cause:** `postinstall: "electron-builder install-app-deps"` recompiles `better-sqlite3` for Electron's Node.js (MODULE_VERSION 125), breaking `npm run dev` on system Node.js (v25.6.0 = MODULE_VERSION 141).

**Fix:** Replace `postinstall` with a dedicated rebuild step only used before Electron packaging:

```json
{
  "scripts": {
    "postinstall": "",
    "rebuild:electron": "electron-builder install-app-deps",
    "electron:build:mac": "npm run build && npm run rebuild:electron && npm run build:electron && electron-builder --mac"
  }
}
```

## 10. Non-Functional Requirements

### Performance
- Sync should handle 500+ resources without UI freeze (paginated, background)
- Table rendering should stay smooth with 500+ rows (virtualization if needed)
- Database operations use prepared statements and transactions for bulk ops

### Security
- Credentials never stored in plain text in production (macOS Keychain)
- No hardcoded secrets in source code
- `contextIsolation: true` in Electron

### Modularity
- No WordPress-plugin-specific code in core components
- All plugin-specific rendering via `getTabs()` / `getFieldRenderers()`
- All site-specific config via profiles
- Adding support for a new WordPress plugin = creating a new plugin directory, no core changes

### Compatibility
- macOS primary (Intel + Apple Silicon)
- Windows/Linux builds possible but not primary targets
- WordPress 5.9+ (REST API v2)
- Node.js 20+ for development (v25 has ABI issues with Electron)

## 15.5 Required Before Public 1.0 Release

Items in ┬ž15 verify Juggernaut works end-to-end for the PLEXKITS team. The items here are gating for selling the app to outside customers.

### Decouple PLEXKITS Profile from the Build

Currently `lib/profiles/index.ts:13-14` does `import plexkitsProfile from './plexkits-resources.json'` and registers it unconditionally in the `ProfileManager` constructor. Every shipped binary contains PLEXKITS sites, taxonomies, plugins, tab layouts, and views ÔÇö so any customer that installs the public release sees "PLEXKITS Resources" as their workspace, with `resource-type` / `topic` / `intent` taxonomies and PLEXKITS-specific tabs (`features`, `downloads`, `changelog`, etc.). That has to stop before the app ships externally.

**Architecture change:**

1. **Bundle a minimal "Default WordPress" profile** (`lib/profiles/default.json`): no `required_plugins` (or only universally-applicable ones), generic taxonomies (`category`, `post_tag`), `post` and `page` post types, no custom tabs, no custom views. This is what every fresh install sees.
2. **Auto-load profiles from `~/.juggernaut/profiles/*.json`** on startup. Mirror the `~/.juggernaut/site-config.json` convention. The loader should:
   - Scan the directory at startup (in `ProfileManager` constructor or a new `discoverUserProfiles()` step in `init.ts`).
   - Validate each file with the same checks as `loadProfileFromFile()`.
   - Register them in `availableProfiles`.
   - Pick a sane default for `currentProfile` (priority: profile with `is_default: true` Ôćĺ most recently modified Ôćĺ bundled default).
3. **Persist imported profiles to disk.** The `POST /api/profile/import-export` route currently registers in-memory only; it must also write the JSON to `~/.juggernaut/profiles/<profile_id>.json` so the import survives restarts. Add a `DELETE /api/profile/import-export?id=<id>` for removal too.
4. **Stop importing `plexkits-resources.json` in the build.** Remove the static import from `lib/profiles/index.ts`. The PLEXKITS profile becomes a private file you and Kerstin keep in `~/.juggernaut/profiles/plexkits-resources.json` (sync via Dropbox / iCloud / a private git repo ÔÇö it's just JSON).
5. **Build a Profile Switcher UI** (or at minimum a Settings Ôćĺ Profiles page) that lists discovered profiles, lets the user import a JSON file, and switches the active profile. The infrastructure for switching already exists in `ProfileManager.setActiveProfile()` ÔÇö just needs a UI.

**Migration for existing PLEXKITS install (you and Kerstin):**

- On first launch of the public-architecture build, if no `~/.juggernaut/profiles/` dir exists AND the bundled default profile is the only one, prompt: "We've moved profiles. Drop your profile JSON in `~/.juggernaut/profiles/` to continue using your existing setup." Show a button that opens that folder.
- Provide a one-time import path: `cp src/lib/profiles/plexkits-resources.json ~/.juggernaut/profiles/` is enough.

**Files that change:**

- `src/lib/profiles/index.ts` ÔÇö remove static import, add disk discovery
- `src/lib/profiles/default.json` (new) ÔÇö minimal generic profile
- `src/app/api/profile/import-export/route.ts` ÔÇö persist imports to disk
- `src/components/SettingsNav.tsx` + new `app/settings/profiles/page.tsx` ÔÇö switcher UI
- `electron-builder.yml` ÔÇö confirm `data/` exclusion already covers user profile dir (it doesn't ÔÇö userData is separate)
- `docs/v1.0-spec.md` ┬ž14 PLEXKITS audit ÔÇö add "static import in `lib/profiles/index.ts`" once removed

**Why this is gating, not optional:**

- A PLEXKITS-branded public release leaks internal site URLs (`staging.plexkits.com`, `plexkits.com`) and exposes proprietary taxonomy schemas.
- Customers buying the app would have to manually scrub the bundled profile, which is a terrible first-run experience.
- The post-v1 "Profile Marketplace" backlog item depends on this architecture existing first.

**Estimated effort:** ~half a day to a day for the core split + persistence; another half-day for a basic switcher UI.

### Tab Layout: Repeater Sub-Field Editor

The Tab Layout settings page (`/settings/tab-layout`) lets users visually assign fields to tabs, but it treats **repeater fields as opaque** ÔÇö you can add `download_sections` to a tab and set its type to `repeater`, but there's no UI to configure the nested sub-fields within a repeater.

**Current behavior:**

- WordPress discovery (`discoverFieldsForPostType()`) returns flat top-level `meta_box` keys. A repeater like `download_sections` appears as a single key because WordPress stores the entire nested structure as one serialized blob.
- The Tab Layout editor's `MappableField` type only has `key`, `label`, `category`, `type` ÔÇö no `fields` sub-property.
- When a user adds a field from the available fields list, `TabLayoutEditor.handleAddField` creates a bare `FieldDefinition` with `type: 'text'` and no sub-fields. The user can change the type to `repeater` via dropdown, but has no way to define sub-fields.
- The actual sub-field schema lives in the profile JSON's `field_layout` under the repeater's `fields` property (e.g., `ui.field_layout.downloads[].fields`). The `RepeaterRenderer` component reads this at render time and recursively renders sub-fields. See `plexkits-resources.json` for the existing example.

**What needs to be built:**

1. **Sub-field editor UI** ÔÇö When a field's type is set to `repeater` in the Tab Layout editor, show a nested panel/modal for defining sub-fields. Each sub-field needs: `key` (the meta_box key), `label`, `type` (text, select, color, checkbox, url, repeater for nesting), and `width` (full, half, quarter).
2. **Conditional visibility** ÔÇö Sub-fields can have a `conditional` property (`{ field, operator, value }`) that controls show/hide based on sibling field values. The editor should support configuring this.
3. **Select options** ÔÇö Sub-fields of type `select` need either static `options` (array of `{ value, label }`) or a `taxonomy_source` (slug of a taxonomy to pull terms from). The editor should support both.
4. **Nested repeaters** ÔÇö The system already supports repeater-within-repeater (e.g., `download_links` inside `download_sections`). The editor should allow at least 2 levels of nesting.
5. **Field discovery hints** ÔÇö Ideally, the editor could inspect the actual `meta_box` data shape from a sample post to suggest sub-field keys and types, rather than requiring manual entry.

**Key files:**
- `src/components/TabLayoutEditor.tsx` ÔÇö The editor component that needs the sub-field UI
- `src/components/fields/RepeaterRenderer.tsx` ÔÇö The runtime renderer that consumes the `fields` sub-structure
- `src/lib/plugins/types.ts` ÔÇö `FieldDefinition` interface (already supports `fields?: Record<string, FieldDefinition>`)
- `src/app/api/tab-layout/route.ts` ÔÇö The API route; `handleAddField` would need to support saving nested structures
- `src/lib/profiles/plexkits-resources.json` ÔÇö Reference example of a fully configured repeater with nested sub-fields (see `ui.field_layout.downloads`)

**Until this is built**, users must author repeater sub-field schemas directly in their profile JSON file. The `plexkits-resources.json` file serves as a reference for the expected structure.
