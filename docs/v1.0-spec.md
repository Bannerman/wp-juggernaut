## 1. Executive Summary

### What Is Juggernaut?

A **modular, plugin-based desktop application** for managing WordPress content. It syncs WordPress posts to a local SQLite database, lets you bulk-edit them offline with a powerful table/modal interface, and pushes changes back with conflict detection. A plugin system extends support for WordPress ecosystem plugins (MetaBox, SEOPress, ACF, Yoast, WooCommerce, etc.).

### Fresh Start vs Fix?

**Recommendation: Fix the existing app.** Here's why:

#### What's Already Solid (~60-70% done)

#### Why Not Start Fresh?

Starting fresh means rewriting **~9,500 lines** and losing 2-3 weeks to get back to the current state. The target architecture is essentially what's already partially built. The plugin type system alone (696 lines of carefully designed interfaces) would take significant effort to recreate.

The broken parts are concentrated in the **UI layer** — specifically 3 component files and 2 config files. The fix path is clear: make UI components consume the plugin/profile system that already exists but isn't wired up to the frontend.

### Key Principle: The Core Knows Nothing About Specific WordPress Plugins

The core app handles: posts, pages, custom post types, taxonomies, basic fields (title, content, excerpt, featured image, status, slug), local SQLite caching, dirty-state tracking, push/pull sync.

**Everything else comes from plugins.** MetaBox fields, SEOPress SEO data, ACF groups, WooCommerce products — all via the plugin system.

## 3. Plugin System

### 3.1 Plugin Interface

Already defined in `src/lib/plugins/types.ts`. Every plugin implements `JuggernautPlugin`:

```typescript
interface JuggernautPlugin {
  // Metadata
  id: string;
  name: string;
  version: string;
  manifest: PluginManifest;

  // Lifecycle
  initialize(core: CoreAPI): Promise<void>;
  activate(profile: SiteProfile, settings: Record<string, unknown>): Promise<void>;
  deactivate(): Promise<void>;

  // Data transformation (WP ↔ Local)
  transformResourceForSync?(resource: WPResource): Promise<WPResource>;
  transformResourceForPush?(resource: LocalResource, payload: PushPayload): Promise<PushPayload>;

  // UI Extensions — THIS IS THE KEY MISSING PIECE
  getTabs?(): TabDefinition[];
  getFieldRenderers?(): Record<string, ComponentType<FieldRendererProps>>;
  getSettingsPanel?(): ComponentType<SettingsPanelProps>;
  getFilterComponents?(): ComponentType<FilterComponentProps>[];

  // WordPress detection
  detectWordPressPlugin?(baseUrl: string, authHeader: string): Promise<boolean>;
}
```

### 3.2 What's Implemented vs What's Missing

| Capability | Interface Defined | Backend Implemented | Frontend Wired Up |
|------------|:-:|:-:|:-:|
| Plugin lifecycle (init/activate/deactivate) | ✅ | ✅ | N/A |
| Data transform hooks (sync/push) | ✅ | ✅ | N/A |
| `getTabs()` — plugin-provided UI tabs | ✅ | ❌ Not called | ❌ EditModal ignores it |
| `getFieldRenderers()` — generic field rendering | ✅ | ❌ Not called | ❌ Fields hardcoded |
| `getSettingsPanel()` | ✅ | ❌ Not called | ❌ |
| `getFilterComponents()` | ✅ | ❌ Not called | ❌ |
| WordPress plugin detection | ✅ | ✅ MetaBox only | ❌ No first-launch wiring |

**The type system is there. The backend hooks work. The frontend doesn't use any of the UI extension points.** This is the core gap.

Each plugin provides its tab components via `getTabs()`:

```typescript

    { id: 'content', label: 'Content', component: MetaBoxContentTab, position: 2 },
    { id: 'features', label: 'Features', component: MetaBoxFeaturesTab, position: 4 },
```

- No hardcoded slugs anywhere

#### CreateModal (Profile-Driven)

- Editable taxonomies come from `profile.taxonomies.filter(t => t.editable)`
- No hardcoded `EDITABLE_TAXONOMIES` array

## 7. Electron Integration

### 7.1 Current Architecture

```
Electron Main Process
├── Creates BrowserWindow (hiddenInset titlebar)
├── Starts Next.js server via utilityProcess.fork()
├── Handles Keychain credentials (safeStorage)
├── Auto-update via electron-updater
└── IPC handlers: get/set/delete credentials, app version, updates
```

### 7.2 Known Issue: Native Module ABI Mismatch

**Root cause:** `postinstall: "electron-builder install-app-deps"` recompiles `better-sqlite3` for Electron's Node.js (MODULE_VERSION 125), breaking `npm run dev` on system Node.js (v25.6.0 = MODULE_VERSION 141).

**Fix:** Replace `postinstall` with a dedicated rebuild step only used before Electron packaging:

```json
{
  "scripts": {
    "postinstall": "",
    "rebuild:electron": "electron-builder install-app-deps",
    "electron:build:mac": "npm run build && npm run rebuild:electron && npm run build:electron && electron-builder --mac"
  }
}
```

### 7.3 Credential Flow

```
┌─────────────────────────────────────────────────────────────┐
│ ELECTRON MODE (Production)                                  │
│                                                             │
│  Settings UI                                                │
│    └─► window.electronAPI.setCredentials()                  │
│          └─► IPC → main process                             │
│                └─► safeStorage.encryptString() → Keychain   │
│                └─► Injects into process.env                 │
│                └─► Restarts Next.js server                  │
│                                                             │

│    └─► wp-client.ts → getWpCredentials()                    │
│          └─► site-config.json credentials (if set)          │
│          └─► process.env fallback (from Keychain injection) │
```

## 15.5 Required Before Public 1.0 Release

Items in §15 verify Juggernaut works end-to-end for the PLEXKITS team. The items here are gating for selling the app to outside customers.

### Decouple PLEXKITS Profile from the Build

Currently `lib/profiles/index.ts:13-14` does `import plexkitsProfile from './plexkits-resources.json'` and registers it unconditionally in the `ProfileManager` constructor. Every shipped binary contains PLEXKITS sites, taxonomies, plugins, tab layouts, and views — so any customer that installs the public release sees "PLEXKITS Resources" as their workspace, with `resource-type` / `topic` / `intent` taxonomies and PLEXKITS-specific tabs (`features`, `downloads`, `changelog`, etc.). That has to stop before the app ships externally.

**Architecture change:**

1. **Bundle a minimal "Default WordPress" profile** (`lib/profiles/default.json`): no `required_plugins` (or only universally-applicable ones), generic taxonomies (`category`, `post_tag`), `post` and `page` post types, no custom tabs, no custom views. This is what every fresh install sees.
2. **Auto-load profiles from `~/.juggernaut/profiles/*.json`** on startup. Mirror the `~/.juggernaut/site-config.json` convention. The loader should:
   - Scan the directory at startup (in `ProfileManager` constructor or a new `discoverUserProfiles()` step in `init.ts`).
   - Validate each file with the same checks as `loadProfileFromFile()`.
   - Register them in `availableProfiles`.
   - Pick a sane default for `currentProfile` (priority: profile with `is_default: true` → most recently modified → bundled default).
3. **Persist imported profiles to disk.** The `POST /api/profile/import-export` route currently registers in-memory only; it must also write the JSON to `~/.juggernaut/profiles/<profile_id>.json` so the import survives restarts. Add a `DELETE /api/profile/import-export?id=<id>` for removal too.
4. **Stop importing `plexkits-resources.json` in the build.** Remove the static import from `lib/profiles/index.ts`. The PLEXKITS profile becomes a private file you and Kerstin keep in `~/.juggernaut/profiles/plexkits-resources.json` (sync via Dropbox / iCloud / a private git repo — it's just JSON).
5. **Build a Profile Switcher UI** (or at minimum a Settings → Profiles page) that lists discovered profiles, lets the user import a JSON file, and switches the active profile. The infrastructure for switching already exists in `ProfileManager.setActiveProfile()` — just needs a UI.

**Migration for existing PLEXKITS install (you and Kerstin):**

- On first launch of the public-architecture build, if no `~/.juggernaut/profiles/` dir exists AND the bundled default profile is the only one, prompt: "We've moved profiles. Drop your profile JSON in `~/.juggernaut/profiles/` to continue using your existing setup." Show a button that opens that folder.
- Provide a one-time import path: `cp src/lib/profiles/plexkits-resources.json ~/.juggernaut/profiles/` is enough.

**Files that change:**

- `src/lib/profiles/index.ts` — remove static import, add disk discovery
- `src/lib/profiles/default.json` (new) — minimal generic profile
- `src/app/api/profile/import-export/route.ts` — persist imports to disk
- `src/components/SettingsNav.tsx` + new `app/settings/profiles/page.tsx` — switcher UI
- `electron-builder.yml` — confirm `data/` exclusion already covers user profile dir (it doesn't — userData is separate)
- `docs/v1.0-spec.md` §14 PLEXKITS audit — add "static import in `lib/profiles/index.ts`" once removed

**Why this is gating, not optional:**

- A PLEXKITS-branded public release leaks internal site URLs (`staging.plexkits.com`, `plexkits.com`) and exposes proprietary taxonomy schemas.
- Customers buying the app would have to manually scrub the bundled profile, which is a terrible first-run experience.
- The post-v1 "Profile Marketplace" backlog item depends on this architecture existing first.

**Estimated effort:** ~half a day to a day for the core split + persistence; another half-day for a basic switcher UI.

### MCP `create_post` + AI Fill Workflow

The MCP server (`src/mcp-server/index.ts`) currently exposes read + update tools only — `list_posts`, `get_post`, `update_post`, `update_seo`, `list_terms`, `update_post_terms`, `get_stats`, `get_post_history`, `get_site_index`. There is no way for an external AI agent (Claude Desktop, Cursor, ChatGPT MCP clients, etc.) to create a new post or to run the AI Fill workflow that the in-app AI Fill tab provides. This backlog item closes both gaps in one coordinated design.

**Goal**

Let an external MCP client (which is itself an LLM) create a post on WordPress and then populate its fields by running the same prompt templates that `AIFillTab` uses today. The handshake mirrors what `AIFillTab` already does — emit a fully-templated prompt, accept a structured response with section markers — except the response comes from the MCP client's own LLM instead of being pasted in by a human.

**New MCP tools (4)**

1. **`create_post`** — mirrors the in-app Create button
   - Input: `{ post_type, title, slug?, status?, content?, excerpt?, featured_media?, meta?, taxonomies? }`
   - Calls `createResource()` in `src/lib/wp-client.ts` (REST POST to WP), then `saveResource()` in `src/lib/sync.ts:268-353` to mirror locally with `is_dirty=0`
   - Returns the new post with real WP ID + meta + terms so the agent can chain follow-up calls
   - Matches `src/app/api/resources/create/route.ts:1-52` behavior exactly so MCP-created posts are indistinguishable from button-created ones

2. **`list_prompt_templates`** — discovery
   - Returns `[{ id, name, description }]` for `ai-fill`, `faq`, `featured-image` plus any user-customized templates registered through `getTemplate()` in `src/lib/prompt-templates.ts:257-285`
   - Lets the agent decide which prompt is relevant for the task it's been given

3. **`get_fill_prompt`** — build the fully-templated prompt for a post
   - Input: `{ post_id, template_id }`
   - Reads the post + meta + terms + available taxonomy terms from SQLite
   - Injects template variables exactly like `AIFillTab.tsx:114-430`: `{{title}}`, `{{intro_text}}`, `{{text_content}}`, `{{terms_<taxonomy>}}`, `{{available_taxonomies}}`, `{{post_taxonomies}}`, `{{features}}`, `{{changelog}}`, `{{downloads}}`, `{{timer_*}}`, `{{seo_*}}`, `{{og_*}}`, `{{twitter_*}}`, `{{faq_items}}`
   - Returns `{ prompt, expected_response_format }` — `prompt` is the full text ready to send to the agent's own LLM; `expected_response_format` documents the section-marker structure the response must use

4. **`apply_fill_response`** — parse structured AI output and write fields
   - Input: `{ post_id, response, template_id? }`
   - Parses section markers `---TITLE---`, `---INTRO_TEXT---`, `---FEATURES---`, `---TAXONOMIES---`, `---TIMER---`, `---DOWNLOADS---`, `---CHANGELOG---`, `---SEO---`, `---SOCIAL---`, `---FAQ_ITEMS---`, `---END---`
   - Resolves taxonomy term names → IDs (same lookup `AIFillTab` does today); resolves file-format names for download links
   - Writes through existing paths: title/content to `posts`, meta to `post_meta` (JSON-encoded), terms to `post_terms`; sets `is_dirty=1` so the next push sync sends it
   - Returns `{ updated_fields: [...], change_count, post }`

**Required refactor (do first, no behavior change)**

Extract two pure modules from `AIFillTab.tsx` so the MCP server and the UI share one source of truth — otherwise we'd be maintaining two parsers and two variable-injection blocks that need to stay in lockstep:

- `src/lib/prompt-renderer.ts` — variable-injection logic currently at `AIFillTab.tsx:114-430`
- `src/lib/ai-fill-parser.ts` — section-marker parser currently at `AIFillTab.tsx:444-797`

Then `AIFillTab.tsx` imports them (no UI change, just an extraction) and the MCP server imports them. This guarantees the in-app and MCP-driven AI Fill produce identical results from the same input.

**Expected agent workflow**

```
list_terms(taxonomy)                       # discover taxonomies + existing terms
create_post(post_type, title)              # stub on WP, get real post_id
get_fill_prompt(post_id, 'ai-fill')        # templated prompt
[agent runs prompt through its own LLM]
apply_fill_response(post_id, response)     # writes title, meta, taxonomies, SEO
get_fill_prompt(post_id, 'faq')            # FAQ pass (optional)
apply_fill_response(post_id, response)
get_fill_prompt(post_id, 'featured-image') # image prompt (optional)
# user reviews in app, push happens normally
```

**Why two-step (prompt out → response in) over server-side LLM**

- The MCP *client* is already an AI. Wiring OpenAI/Anthropic into the MCP server would be a redundant second LLM integration — and would require API key management in a process that has deliberately stayed AI-provider-agnostic.
- The in-app `AIFillTab` already uses the same handshake: emit prompt, parse pasted response. Reusing the parser keeps formats consistent and means user edits to prompts apply to both the UI and MCP agents.
- The agent picks its own model and budget. The Juggernaut MCP surface stays portable across any MCP-speaking client.

**Open questions to resolve at build time**

1. `create_post` behavior: hit WP immediately (matches button) or offer `draft_only: true` for local-first stubs that push later creates? Lean *match the button* — agent gets a real ID to chain. A `draft_only` flag could be added later if the use case emerges.
2. `apply_fill_response` behavior: mark `is_dirty=1` and let the next push sync (matches `update_post`) or push immediately? Lean *mark dirty only* — keeps the model "agent stages, user reviews, user pushes". If agents need to push, they can call a separate `push_post` tool (not currently exposed; would be a fifth addition).
3. Prompt editing via MCP: read-only, or let agents edit `getTemplate()` content too? Lean *read-only* — user owns prompts in the app. Agents shouldn't rewrite shared infrastructure mid-flow.
4. Should `create_post` also accept a `seed_with_ai_fill: true` shortcut that internally runs the get-prompt → apply cycle? Probably no — better to keep tools single-purpose so the agent can inspect the prompt and decide.

### Tab Layout: Repeater Sub-Field Editor

The Tab Layout settings page (`/settings/tab-layout`) lets users visually assign fields to tabs, but it treats **repeater fields as opaque** — you can add `download_sections` to a tab and set its type to `repeater`, but there's no UI to configure the nested sub-fields within a repeater.

**Current behavior:**

- WordPress discovery (`discoverFieldsForPostType()`) returns flat top-level `meta_box` keys. A repeater like `download_sections` appears as a single key because WordPress stores the entire nested structure as one serialized blob.
- The Tab Layout editor's `MappableField` type only has `key`, `label`, `category`, `type` — no `fields` sub-property.
- When a user adds a field from the available fields list, `TabLayoutEditor.handleAddField` creates a bare `FieldDefinition` with `type: 'text'` and no sub-fields. The user can change the type to `repeater` via dropdown, but has no way to define sub-fields.
- The actual sub-field schema lives in the profile JSON's `field_layout` under the repeater's `fields` property (e.g., `ui.field_layout.downloads[].fields`). The `RepeaterRenderer` component reads this at render time and recursively renders sub-fields. See `plexkits-resources.json` for the existing example.

**What needs to be built:**

1. **Sub-field editor UI** — When a field's type is set to `repeater` in the Tab Layout editor, show a nested panel/modal for defining sub-fields. Each sub-field needs: `key` (the meta_box key), `label`, `type` (text, select, color, checkbox, url, repeater for nesting), and `width` (full, half, quarter).
2. **Conditional visibility** — Sub-fields can have a `conditional` property (`{ field, operator, value }`) that controls show/hide based on sibling field values. The editor should support configuring this.
3. **Select options** — Sub-fields of type `select` need either static `options` (array of `{ value, label }`) or a `taxonomy_source` (slug of a taxonomy to pull terms from). The editor should support both.
4. **Nested repeaters** — The system already supports repeater-within-repeater (e.g., `download_links` inside `download_sections`). The editor should allow at least 2 levels of nesting.
5. **Field discovery hints** — Ideally, the editor could inspect the actual `meta_box` data shape from a sample post to suggest sub-field keys and types, rather than requiring manual entry.

**Key files:**
- `src/components/TabLayoutEditor.tsx` — The editor component that needs the sub-field UI
- `src/components/fields/RepeaterRenderer.tsx` — The runtime renderer that consumes the `fields` sub-structure
- `src/lib/plugins/types.ts` — `FieldDefinition` interface (already supports `fields?: Record<string, FieldDefinition>`)
- `src/app/api/tab-layout/route.ts` — The API route; `handleAddField` would need to support saving nested structures
- `src/lib/profiles/plexkits-resources.json` — Reference example of a fully configured repeater with nested sub-fields (see `ui.field_layout.downloads`)

**Until this is built**, users must author repeater sub-field schemas directly in their profile JSON file. The `plexkits-resources.json` file serves as a reference for the expected structure.
